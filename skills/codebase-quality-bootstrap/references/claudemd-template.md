# CLAUDE.md Template Structure

This is the canonical template for generated CLAUDE.md files. Each section maps to one or more codebase-audit domains. Sections marked `[CONDITIONAL]` are only included when the corresponding tech is detected.

## Line Budget

- **Target:** 120-180 lines when populated
- **Hard limit:** 200 lines
- **Overflow strategy:** Condense rules into tables, merge related items, or move detailed rules to `.claude/quality-rules.md` referenced via `@.claude/quality-rules.md`

## Standard Section Headings

These headings are recognized as STANDARD sections during brownfield analysis. Any heading not in this list is classified as CUSTOM and preserved verbatim.

```markdown

| `{test_cmd}` | Run all tests |
| `{lint_cmd}` | Lint code |
| `{format_cmd}` | Format code |

Only include commands that actually exist in the project. Omit rows for commands that are not configured.

## Architecture

```
{detected directory structure with purpose annotations}
```

Brief annotation of module boundaries and dependency direction. 5-15 lines max.

## Code Standards

{QUAL rules: tech-stack-specific quality rules}
{DEAD rules: dead code prevention}
{DEPR rules: deprecated pattern avoidance}

Rules must reference the detected language/framework. Example:
- Good: "Use `Optional<T>` instead of nullable returns in all public Java methods"
- Bad: "Handle nulls properly"

### Enterprise Mandates

- All code uses current, officially recommended APIs
- No migration scripts, compatibility layers, or transition logic
- Current best practices applied consistently to every component
- No backward compatibility shims, version checks, or legacy adapters
- No "old/new/legacy" labeling -- everything is the current state
- No partial patches preserving outdated structures
- No assumptions about pre-existing users, data, or state

These 7 mandates are always included verbatim.

## Security

{SEC rules: OWASP-aligned security rules}
{PRIV rules: data privacy rules}

Must include at minimum:
- Parameterized query requirement (with framework-specific syntax)
- Input validation at boundary
- Secrets management (env vars, never hardcoded)
- PII handling (no PII in logs, URLs, error responses)
- Authentication/authorization pattern (framework-specific)

## Architecture Rules

{ARCH rules: dependency direction, module boundaries}

Must include at minimum:
- Dependency flow direction
- Circular dependency prohibition
- Module responsibility principle

## Concurrency

[CONDITIONAL: Only include if Phase 1 detects threading, goroutines, async runtimes, multiprocessing, or actor systems]

{CONC rules: shared state, synchronization, async patterns}

## Testing

{TEST rules: test requirements, coverage, assertions}

Must include at minimum:
- Test file naming convention
- Test commands (full suite + single file)
- Assertion quality requirement
- Mock usage guidelines

## Dependencies

{DEP rules: dependency management}

Must include at minimum:
- Version pinning requirement
- Lock file commitment
- No unused dependencies

## Infrastructure

[CONDITIONAL: Only include if Dockerfile, docker-compose, CI/CD configs, or IaC detected]

{INFRA rules: container best practices, CI/CD, env var hygiene}

## Performance

{PERF rules: query patterns, bounded collections, resource management}

Must include at minimum:
- N+1 query prevention
- Collection bounding (pagination, limits)
- Resource lifecycle management

## Documentation

{DOC rules: what must be documented}

Must include at minimum:
- README accuracy requirement
- Public API documentation
- Configuration documentation

## Gotchas

{Project-specific non-obvious patterns detected during Phase 1 reconnaissance}

Only include if genuinely non-obvious patterns were detected. Do not fabricate gotchas.
```
